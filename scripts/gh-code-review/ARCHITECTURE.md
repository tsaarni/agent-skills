# Deterministic Code Review Assistant - Architecture

This document describes the architecture of the `gh-code-review` tool.

## Overview

The program automates the preparation of a structured review environment for AI agents. It supports reviewing GitHub Pull Requests as well as local work-in-progress (unpushed commits, staged, and unstaged changes). It extracts what code has changed and how those changes impact the broader repository.

---

## 1. The Extraction Engine

The core of the tool consists of an extraction engine that collects and synthesizes all relevant information about the code changes:

1.  **The Diff (Git)**
    - The definitive code changes.
    - **Unified Diff Engine**: Uses local `git diff -W` against the common ancestor (`git merge-base`) of the base branch and HEAD. This ensures consistent function context and includes unpushed commits as well as staged/unstaged changes.
2.  **The Metadata (GitHub)**
    - **PR Enhancement**: If the current branch is linked to an active PR (or if a PR is explicitly targeted), detailed PR metadata is fetched via **GitHub GraphQL API** (using `gh api graphql`). This includes the PR title, description, comments, and review threads (with file/line context) to provide the LLM with the full discussion history.
3.  **Context Extraction**:
    - Uses **Git's native `userdiff` drivers** (via hunk headers) and the `-W` (`--function-context`) flag to identify the context enclosing changed lines.
4.  **Impact Analysis (Semantic)**:
    - The identifier extraction is performed **semantically** by the LLM during the initial phase of the review.
    - The LLM identifies modified symbols (functions, classes) that experience a signature or behavioral change.
    - The agent then performs a repository-wide search to find and verify the callers of those specific symbols.

---

## 2. Output Artifacts

The engine produces a structured directory (e.g., `gh-code-review/123/` or `gh-code-review/local-<branch>/`) containing:

- **`pr.diff`**: The raw unified Git diff (including function context).
- **`metadata.json`**: Title, description, and metadata (from GitHub).
- **`diff-with-function-context.json`**: JSON-wrapped source code for only the modified functions/types. Each hunk includes line numbers and a `*` marker for added lines.
- **`agent-orchestrator.prompt.md`**: A specialized prompt for the **3-Stage Review Execution Plan**.

In the parent directory (e.g., `gh-code-review/`):

- **`review.prompt.md`**: A static set of high-assurance review stages and noise filters.

---

## 3. Review Execution

The goal is to provide multi-phase execution plan to reduce "noisy" single-shot reviews while still applying the full comprehensive review.

### Phase 1: Subagent 1 ("Analyze Impact")

- **Focus**: Map the blast radius of the changes without making qualitative judgments.
- **Task**: Identify modified public symbols (functions, classes, interfaces, types) and use codebase search tools to find their external callers.
- **Output**: `gh-code-review/123/results/impact-analysis.json`.

### Phase 2: Subagent 2 ("Review Code")

- **Focus**: Perform the actual, comprehensive code review.
- **Input**: The raw code, the diff, and the `impact-analysis.json` generated by Agent 1.
- **Task**: Analyze the code for bugs, architectural issues, and integration failures identified in the impact analysis.
- **Output**: `gh-code-review/123/results/review-findings.json`.

### Phase 3: Subagent 3 ("Criticize Results")

- **Focus**: Act as a strict quality filter, critic, and final consolidator.
- **Task**: Critically evaluate each finding from Agent 2 for concrete evidence, relevance, and formatting. If critical gaps or inconsistencies are found, it rejects the findings and triggers a feedback loop to Agent 2 with a `gh-code-review/123/results/critique.md`. If accepted, it deduplicates and generates the final results.
- **Output**: `gh-code-review/123/results/review-results.md` (or triggers a loop).

---

## 4. Hard Rules & Constraints

- **No Speculation**: Findings must be directly observable from the provided code and diff.
- **Exact Referencing**: Every issue must point to a specific file and line/range.
- **Context Isolation**: Subagents are given only the context relevant to their task to prevent context window degradation and noise.
- **Clean Reviews are Valid**: If no concrete issues are found, the output is a clean status report.
